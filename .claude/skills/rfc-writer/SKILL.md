# RFC Writer

프로젝트의 기술적 의사결정을 RFC(Request for Comments) 문서로 체계적으로 기록하는 스킬.

## RFC란 무엇인가

RFC는 프로젝트에 영향을 미치는 중요한 기술적 결정을 제안하고, 논의하고, 기록하는 문서다. 단순한 버그 수정이나 사소한 개선이 아니라, 아키텍처·기술 선택·규약 변경처럼 "되돌리기 어렵고 팀 전체에 영향을 주는" 결정을 다룬다.

RFC의 핵심 가치는 **"왜(Why)"를 보존하는 것**이다. 코드는 "무엇(What)"을 말해주지만, 왜 그 선택을 했는지는 시간이 지나면 사라진다. RFC는 미래의 팀원이 "왜 이렇게 했지?"라고 물었을 때 답을 찾을 수 있는 장소다.

## RFC를 써야 하는 시점

다음 중 하나에 해당하면 RFC를 작성한다:

- 새로운 라이브러리·프레임워크·도구 도입
- 기존 아키텍처의 구조적 변경
- 팀 전체에 적용되는 코딩 컨벤션·규약 변경
- 외부 서비스·API 통합 방식 결정
- 성능·보안·접근성에 영향을 주는 기술적 트레이드오프
- 되돌리기 어려운(hard-to-reverse) 결정

## 파일 위치와 네이밍

```
docs/
└── decisions/
    ├── 0001-use-nextjs-app-router.md
    ├── 0002-adopt-tanstack-query.md
    ├── 0003-monorepo-with-turborepo.md
    └── _template.md
```

**네이밍 규칙:**
- 경로: `docs/decisions/` (프로젝트 루트 기준)
- 파일명: `{4자리 시퀀스}-{kebab-case 제목}.md`
- 시퀀스는 단조 증가하며, 거부(rejected)된 RFC도 번호를 유지한다
- 번호를 재사용하지 않는다

기존 프로젝트에 이미 `docs/rfcs/`, `docs/adr/`, `architecture/decisions/` 같은 경로가 있다면 그것을 따른다. 새 프로젝트라면 `docs/decisions/`를 기본으로 사용한다.

## RFC 작성 워크플로우

### 1단계: 맥락 파악

사용자의 요청에서 다음을 파악한다:

- **무엇을 결정하려는가** — 핵심 질문 1문장으로 정리
- **왜 이 결정이 필요한가** — 현재 상황의 문제점이나 새로운 요구사항
- **어떤 선택지가 있는가** — 최소 2개 이상의 대안
- **제약 조건** — 일정, 팀 역량, 기존 인프라, 비용 등

사용자가 충분한 맥락을 제공하지 않았다면, 위 항목을 질문해서 채운다. 추측으로 채우지 않는다.

### 2단계: RFC 문서 작성

`references/rfc-template.md`를 읽고 템플릿 구조를 따른다. 핵심 원칙:

- **Context 섹션**: 가치 중립적 서술. 사실만 기술하고, 의견을 섞지 않는다
- **Decision 섹션**: 능동태로 "우리는 ~한다(We will ...)" 형식으로 작성한다
- **Consequences 섹션**: 긍정적·부정적·중립적 결과를 모두 기술한다. 장점만 나열하지 않는다
- **Alternatives 섹션**: 기각된 대안도 왜 기각했는지 이유를 명시한다

### 3단계: 번호 할당

기존 RFC 파일들을 확인해서 다음 시퀀스 번호를 사용한다:

```bash
ls docs/decisions/ | grep -E '^[0-9]{4}-' | sort -r | head -1
```

기존 파일이 없으면 `0001`부터 시작한다.

### 4단계: 파일 생성 및 검증

- Markdown 파일을 `docs/decisions/` 에 생성한다
- 필요하다면 `docs/decisions/` 디렉토리도 함께 생성한다
- 작성 후 문서를 한번 읽고, 다음을 확인한다:
  - 모든 섹션이 채워져 있는가
  - Context가 사실 기반인가 (의견이 섞이지 않았는가)
  - Decision이 명확하고 실행 가능한가
  - Consequences에 부정적 영향도 포함되어 있는가
  - 대안이 최소 1개 이상 기술되어 있는가

## RFC 상태(Status) 관리

| Status | 의미 |
|--------|------|
| `proposed` | 제안됨. 논의 중 |
| `accepted` | 승인됨. 구현 예정 |
| `rejected` | 거부됨. 사유 기록 |
| `deprecated` | 과거에 승인되었으나, 새 RFC로 대체됨 |
| `superseded` | 다른 RFC로 완전히 대체됨. 대체 RFC 번호를 명시 |

새 RFC의 기본 상태는 `proposed`다. 사용자가 명시적으로 상태를 지정하면 그것을 따른다.

기존 RFC를 대체하는 경우:
1. 새 RFC에 "Supersedes: RFC-XXXX" 를 기재한다
2. 기존 RFC의 상태를 `superseded`로 변경하고, "Superseded by: RFC-YYYY"를 추가한다

## 작성 스타일 가이드

**어조**: 미래의 개발자와 대화한다고 생각하고 쓴다. 지나치게 형식적일 필요 없지만, 모호하면 안 된다.

**분량**: 1~2페이지(Markdown 기준 50~150줄). 길어질수록 읽히지 않는다. 핵심에 집중한다.

**기술 용어**: 팀 내에서 통용되는 용어를 사용하되, 외부에서 온 새 팀원이 이해할 수 있도록 약어는 처음 등장할 때 풀어 쓴다.

**코드 예시**: 결정의 영향을 보여주는 최소한의 코드만 포함한다. RFC는 구현 명세가 아니다.

**링크**: 관련 이슈, PR, 외부 문서에 대한 링크를 적극 활용한다. 맥락이 풍부할수록 좋다.

## 한국어 RFC 작성 시 주의사항

- 기술 용어(React, Server Component, API 등)는 원어 그대로 사용한다
- 섹션 제목은 영어로 유지한다 (Context, Decision, Consequences 등) — 다국어 팀에서의 호환성을 위해
- 본문은 한국어로 작성하되, 코드 블록 내 주석은 영어를 권장한다
- "~하기로 한다", "~를 채택한다" 같은 결정 문체를 사용한다
